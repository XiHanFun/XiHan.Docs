# 权限模型

XiHan.BasicApp 的权限体系融合了 **RBAC（基于角色）** 与 **ABAC（基于属性）**，并叠加数据范围、字段级脱敏与多租户隔离。这是系统最核心也最需要理解的部分。

## 权限码

权限的基本单位是**权限码**，格式为三段：

```text
resource : action : scope
资源       动作      范围
```

- 例：`user:read:dept`（读取本部门用户）、`order:export:self`（导出本人订单）
- 超级管理员使用通配 `*:*:*`，放行一切

前端页面、字段、操作三级都按权限码过滤：没有对应权限码，元素不渲染 / 不可操作。

## RBAC：角色驱动

- 用户通过**角色**获得权限码集合
- 角色支持**层级继承**（用闭包表存储层级关系），子角色继承父角色权限
- **会话角色激活**：登录后可在一次会话内激活/切换生效角色，实现动态职责分离（同一用户不同时刻以不同角色办事）

## 数据范围（Data Scope）

同一个权限码，能看到的数据行随**数据范围**收窄：

| 范围 | 含义 |
| --- | --- |
| 本人（self） | 只能操作自己的数据 |
| 部门（dept） | 可操作本部门（含子部门）数据 |
| 租户（tenant） | 可操作本租户全部数据 |

数据范围在查询层生效，与权限码组合决定"能对哪些行做这个动作"。

## 字段级脱敏（FLS）

即使有权读取某条记录，**敏感字段**也可能被脱敏（如手机号、证件号打码）。脱敏由字段级权限门控：可读且未脱敏才显示原值，否则显示掩码。列表页的导出与显示保持一致的脱敏策略。

## ABAC：属性驱动的约束

在 RBAC 之上，ABAC 用**属性与规则**做更细的运行时约束：

- **约束规则**：时间窗（仅工作时间可操作）、IP（仅内网可访问）、表达式（自定义条件）
- 评估时综合主体属性（用户）、资源属性、环境属性，返回是否放行

> 框架侧的授权采用**实时校验**而非信任令牌里的冻结声明——权限的授予/回收、用户禁用、会话失效都会立即生效。

## 权限申请与审批

- 用户可发起**权限申请**，走审批流程
- 权限的每次变更**留痕**（谁、何时、授予/回收了什么），便于审计追溯

## 多租户隔离

- **字段级隔离**：数据按 `TenantId` 隔离，全局数据使用 `TenantId=0` 约定
- **邮箱全局唯一登录**：登录后按用户归属自动落点（平台控制台 / 租户工作台 / 租户选择页），可随时切换租户
- **超级管理员平台态运维**：可切入任意租户代为管理
- **租户版本（Edition）门控**：不同版本对应不同的权限白名单，运行时门控；开通版本时一站式创建管理员、角色与授权；**降级时自动回收越权授权**

## 组合起来

一次访问的判定链大致是：

```text
请求进来
  → 认证：你是谁（JWT 解析出用户/租户/会话）
  → 租户解析：落到哪个租户上下文
  → 授权(RBAC)：有没有这个权限码
  → 授权(ABAC)：属性约束是否满足（时间/IP/表达式）
  → 数据范围：能看到哪些行
  → 字段脱敏：敏感字段是否显示原值
```

每一层都可能把访问收窄或拒绝，层层收紧，默认从严。
